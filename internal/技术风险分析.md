# 技术风险分析 · 深度思考

## 1. OOC校验分数公式

**核心问题**：分数不能拍脑袋算。评审看到"88% ✅"这个数字，应该觉得它背后有严谨的统计学基础，而不是"哦，LLM自己打了个分"。

### 设计思路：多因子模型

每个选项的 OOC 风险评分 = 多个可测量维度的加权合成。

**注意：P（惊奇度）权重需要调大一些，实际需要在工程阶段反复调试。** 因为P是反直觉因子——角色出乎意料但合理的发展才是好角色。它的权重太小的话，公式就永远奖励"最安全但最无聊"的选项。

#### 因子定义

| 因子 | 符号 | 含义 | 心理学对应 |
|------|------|------|-----------|
| **特质一致性** | T | 选项是否违背角色的中心特质（Asch的中心特质） | Asch Configural Model |
| **行为一致性** | B | 选项在类似历史情境下是否有先例 | Bartlett Schema Theory |
| **语义距离** | D | 选项的语义向量与角色语义记忆"重心"的距离 | Tulving Semantic Memory |
| **自洽性** | C | 选项内部是否有逻辑矛盾 | Kintsch CI Model |
| **惊奇度** | P | 选项的不可预测程度（作为创造性指标） | 对抗"太安全"的bias |

#### 主公式（初版）

```
OOC_Risk = 1 - (αT + βB + γ(1-D) + δC - εP)
```

其中：
- α + β + γ + δ + ε = 1（权重之和为1）
- T, B, D, C, P 均为归一化到 [0,1]
- **OOC_Risk 越高 → 越可能OOC**（输出为百分比）

#### 各因子具体计算方法

**T（特质一致性）** — 硬约束
- 角色档案中有标记为"中心特质"的条目（如：Leo的核心特质 = 忠诚、隐忍、保护欲）
- 将选项+情境输入Qwen，给出binary判断：是否违背任一中心特质？
- T = 1（无违背） / 0（有违背）
- **权重 α 应该最高**，因为中心特质被破 → 基本OOC

**B（行为一致性）**
- 从情景记忆中检索top-k最相似历史情境
- 对比角色在那些情境中的选择 / 反应
- B = 相似情境中选择一致的比例
- 如果k=0（无历史先例），B = 0.5（中性，不惩罚也不加分）

**D（语义距离）**
- 对选项做 embedding → 得到向量 v_option
- 角色语义记忆中有"认知重心"向量 v_char（历史所有相关事件 embedding 的质心）
- D = cosine_distance(v_option, v_char)
- 低D = 角色说出来的话跟他平时的"感觉"一致

**C（自洽性）**
- 让Qwen检查选项内部是否有矛盾
- C = 1（自洽） / 0（不自洽）

**P（惊奇度）**
- 这是**反直觉因子**：太符合预期的选项反而创造性不足
- 同样用语义距离，但跟"角色认知模型中**预期值**"对比
- 如果选项完全在预期内 → P=0（不扣分也不加分）
- 如果选项出乎意料但又不违背特质 → P 增加，**降低**OOC_Risk（因为这说明角色有成长/弧光）
- 用来避免"永远输出最安全的选项"的bias

#### 公式结构关系图（文字版）

```
                    ┌──────────────────────┐
                    │     选项 + 情境        │
                    └──────────┬───────────┘
                               │
                ┌──────────────┼──────────────┐
                ▼              ▼              ▼
        ┌──────────┐   ┌──────────┐   ┌──────────┐
        │  T校验    │   │ B检索对比 │   │ D向量距离 │
        │ (hard)   │   │ (soft)   │   │ (fast)   │
        └────┬─────┘   └────┬─────┘   └────┬─────┘
             │              │              │
             ▼              ▼              ▼
         T=0/1           B∈[0,1]         D∈[0,1]
             │              │              │
             └──────┬───────┘              │
                    │                      │
                    ▼                      ▼
               αT + βB              γ(1-D) — 距离越近越好
                    │                      │
                    └──────────┬───────────┘
                               │
                ┌──────────────┼──────────────┐
                │              │              │
                ▼              ▼              ▼
           C自洽校验        +εP(惊奇)      ═══> OOC_Risk
           (旁路)            (反直觉)       (最终输出)
```

- T+B先汇聚，再与D汇聚 → 主干道
- C作为旁路独立校验（δC）
- P从右侧以虚线形式减去（-εP），用于反直觉修正
- T↔P之间存在核心张力：T约束角色不崩，P鼓励弧光成长

#### 公式的美学

这个公式的美在于：
1. **每一项都有心理学对应** → 评审追问你有理论依据
2. **权重可解释** → α > β > γ（特质 > 行为 > 语义），人可以理解为什么
3. **P因子防止过拟合** → 不是追求"角色永不犯错"，而是"角色犯错得合理"
4. **输出带可解释性** → 不只给分数，给明细："特质一致性100%，行为一致性75%，语义距离低(0.12)，自洽性100%，惊奇度中等 → 综合OOC风险12%"

#### 实现上的简化

实际第一版可以做：
- T用LLM打分（binary，准）
- B用embedding检索 + LLM判断（准，但慢一点可接受）
- D直接用embedding距离（快，不用LLM）
- C用LLM（一锤子，简单）
- P用embedding距离的second-moment（统计量，有数学感）

第一次实现不需要完美，但这个**公式结构**要在提交时的方案说明里写清楚。评审看到"OOC风险 = 1 - (αT + βB + γ(1-D) + δC - εP)"，再看到每一项的心理学引用，会觉得这不是糊弄的。

## 2. 情景记忆索引

已确认方案：
- **阅读时**：只做"主角 + 时间"索引 + 事件摘要
- **睡眠巩固时**：提取因果链、意图、关系变化等深层信息

深度思考：主角+时间的存储结构

每个事件记录为：
```json
{
  "id": "evt_0042",
  "chapter": 3,
  "position": "3/4",        // 章节内位置
  "protagonist": "Leo Chen",
  "timestamp_rel": "ch3_after_meeting",
  "summary": "Leo在会议后独自留在房间，翻看旧照片",
  "embedding": [0.12, ...],  // 事件语义向量
  "importance": 0.7,        // 重要性权重（后续用于剪枝）
  "related_entities": ["Li Wei", "old_photo"]
}
```

**关键点**：每个事件的 `importance` 初始值由LLM在阅读时快速评估，后续在睡眠巩固中根据使用频率/关联度调整。低重要性事件是剪枝的候选。

## 3. 长期性能问题

你说得对，核心思想是：**当前的模型够快，未来的模型更快，我们只需要保证架构在合理范围内不退化成不可用**。

### 当前可做的几件事

**逐层缩小搜索域**：
```
用户提问
    ↓
语义检索 → top-50候选事件（embedding近似搜索，<100ms）
    ↓
主角筛选 → 只保留该角色相关（精确过滤，<10ms）
    ↓
重要性排序 → top-10送入LLM做上下文（成本可控）
```

**事件压缩**：
- 超过100个事件后，自动将最早/最低权重的同类事件合并为摘要
- 类似"Leo在第1-5章期间表现出明显的回避倾向，共3次在冲突面前选择退让"

**缓存**：
- 相似问题可直接返回缓存结果（如："Leo是什么样的人"这类元问题）
- 相似度超过阈值的已答问题命中缓存

**这些都不需要在初版做完**，但里程碑里标注为"性能优化周"的内容。评审问起来可以说"我们的架构设计支持渐进式优化，当前版本在N个事件以内响应<X秒"。

## 4. "角色困惑"的魔法时刻

### 问题本质

LLM有一个**完成的本能**——你问它一个问题，它倾向于给出一个完整的、确定的答案。让它"说一半"、"说不清楚"是反本能的。

### 应对策略

**Prompt工程（兜底方案）**：
在角色扮演的system prompt中加入：
```
你的回答不一定要完整。如果你作为[角色名]自己也不确定
某个感受或想法，你可以表达这种不确定性——甚至可以说
"我不知道"或"我说不上来"。真实的角色比完美的角色更可信。
```

**但是**：这个prompt的效果不稳定，80%的情况下LLM还是会给出完整回答。

**Demo策略**：
- Demo录视频时，这个场景走**人工筛选**——多问几次，选到满意的
- 或者，在特定问题下用**手动注入**的方式写死一段响应，同时展示系统"也支持自然生成"
- 在评审问答环节可以说"我们专门优化了prompt让角色可以表达不确定性，因为**真实角色不完美**本身就是我们这个产品的核心理念"

**未来改进方向**（可写在方案中但不实现）：
- 用few-shot示例让角色学会"说一半"
- 更激进：在角色语义记忆中标记"未解决的情感冲突区域" → 当问题落入这些区域时，输出模式切换为"不确定模式"

## 6. OOC 分数置信度（数据稀疏期）

### 问题

公式假设所有因子 T/B/D/C/P 的测量精度相等。但早期数据很少时，各因子置信度差异很大：

| 因子 | 早期表现 | 问题 |
|------|---------|------|
| T（特质一致性） | LLM binary判断，0样本也可用 | 稳定，但LLM可能有bias |
| B（行为一致性） | k=0时返回中性0.5 | "无先例"和"50%一致"是两种含义，混为一谈 |
| D（语义距离） | 单一样本也可计算 | 稳定 |
| C（自洽性） | LLM binary判断 | 同T |
| P（惊奇度） | 有centroid即可算 | 早期centroid不稳定 |

### 当前策略

数据稀疏期不做置信度加权。理由：
1. 黑客松 Demo 场景下，角色已有28章的语义记忆，B因子不会为0
2. 新增角色时，用户会在问角色前先摄入小说（IngestionService），摄入完成后语义记忆已有足够数据
3. 置信度加权增加系统复杂度，边际收益在黑客松阶段不高

### 未来方向（不实现，仅存档）

可增加一个"样本量"维度：T_conf, B_conf, D_conf, C_conf, P_conf ∈ [0,1]，公式改为：

```
OOC_Risk = 1 - (α·T·T_conf + β·B·B_conf + γ·(1-D)·D_conf + δ·C·C_conf - ε·P·P_conf)
```

样本量低的因子自动降权。但需注意：这增加了5个可调参数，调试成本上升。

## 7. 跨会话演示策略

### 问题

`跨会话记忆分析.md` 建议的"关浏览器 → 重开"演示方案在现场不够优雅：
- 等待后端起
- 网络闪一下就穿帮
- 节奏不可控

### 替代方案：切角色法

**核心思路**：跨会话的本质是"离开再回来，记忆还在"。不需要关浏览器，用角色切换来触发同样的 serialize/deserialize 边界：

```
演示流程：
  1. 用户与角色A对话 → 工作记忆积累上下文 → 用户做选择
  2. 用户切到角色B（触发角色A的序列化）
  3. 用户在角色B问一个问题（确认B的记忆独立）
  4. 用户切回角色A（触发角色A的反序列化）
  5. 角色A记得第1步的对话 → 跨会话能力已证明 ✅
```

**优势**：
- 同页面内完成，节奏可控
- 不需重启服务
- 同时展示了"多角色记忆隔离"——这是架构上的加分项
- 演示时长可控在30秒内

### 补充

正式提交前仍可录一段"关浏览器重开"的实机视频作为备选材料，但现场演示以切角色法为主。
